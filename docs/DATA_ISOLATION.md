# ABC-IO v2.0 Data Isolation Guide

## Purpose

This document defines the four-bucket data isolation model used by ABC-IO v2.0. It separates public and private data, internal AI processing from customer content, and sets storage, access, backup, and retention expectations.

## Overview

All platform data is classified into one of four buckets:

| Bucket | Purpose |
|---|---|
| `ai_private` | Internal AI processing, protected prompts, orchestration state |
| `ai_public` | Public AI-facing materials, public assistants, approved generated content |
| `my_private` | Owner, operator, and protected customer/admin content |
| `my_public` | Public business information and approved customer-facing content |

## Bucket definitions

### ai_private

| Attribute | Policy |
|---|---|
| Allowed contents | Internal orchestration state, nonpublic logs, secured service metadata, provider API responses |
| Forbidden contents | Public user content without authorization, secrets in plaintext |
| Exposure policy | No public exposure; access limited to `kimi`, `worker`, `autonomous`, and authorized operators |
| Access policy | Service-to-service only; owner approval for human access |
| Storage policy | `postgres` and ephemeral caches in `redis`; logs in container stdout |
| Backup policy | Encrypted PostgreSQL dumps; exclude transient `redis` queues |
| Retention policy | 30 days for logs; model responses retained per account tier and legal need |
| Examples | `kimi` circuit-breaker state, `worker` job payloads, `autonomous` decision logs |

### ai_public

| Attribute | Policy |
|---|---|
| Allowed contents | Public assistants, public prompts, approved generated content, demo translations |
| Forbidden contents | Secrets, private operator data, unmoderated user prompts |
| Exposure policy | Public via `public-portal`, `account-pwa`, `interface-pwa` |
| Access policy | Anonymous demo rate limits; authenticated users under tier quotas |
| Storage policy | `postgres` for approved content; `redis` cache with TTL |
| Backup policy | Included in standard PostgreSQL backups |
| Retention policy | Until explicitly revoked or superseded |
| Examples | Public help articles, marketing copy generated by AI, demo translations |

### my_private

| Attribute | Policy |
|---|---|
| Allowed contents | Admin records, internal docs, restricted configs, protected account data, direct messages |
| Forbidden contents | Public publication without approval, sharing outside the account |
| Exposure policy | Account-scoped only; encrypted in transit and at rest |
| Access policy | Authenticated users within the same account; owner/operators for admin data |
| Storage policy | `postgres` with row-level account scoping; `redis` only for transient sessions |
| Backup policy | Encrypted PostgreSQL dumps; GPG-encrypted offsite copies |
| Retention policy | Account lifetime plus legal hold; messages retained per conversation lifecycle |
| Examples | `conversations`, `direct_messages`, `subscriptions`, owner dashboard actions |

### my_public

| Attribute | Policy |
|---|---|
| Allowed contents | Marketing pages, help pages, plan descriptions, public status and trust content |
| Forbidden contents | Secrets, private records, internal-only operations data |
| Exposure policy | Public via `nginx` and `public-portal` |
| Access policy | Unauthenticated read; owner approval for writes |
| Storage policy | Static files in `services/public-portal/src/public/`; rendered HTML |
| Backup policy | Source code in Git; release archives |
| Retention policy | Indefinite while published; archived on replacement |
| Examples | Homepage, pricing page, accessibility statement, brand guide |

## Database isolation

- `accounts.id` scopes `users`, `conversations`, `direct_messages`, `subscriptions`, `usage_logs`, and `account_products`.
- `conversation_participants` enforces that participants belong to the same account.
- `family_preferences` provides account-level safe-content controls.

## Owner-gated actions

`ACTION REQUIRED FROM OWNER`
- item needed: approve retention periods and encryption passphrase for offsite PostgreSQL backups
- reason: `my_private` and `ai_private` data require protected backups; retention defines legal and recovery scope
- where it is needed: backup scripts, password manager, and offsite storage location
- exact steps:
  1. Choose a retention policy (default: 7 daily, 4 weekly, 12 monthly).
  2. Generate a strong GPG passphrase and store it in the enterprise password manager.
  3. Update `scripts/backup-postgres.sh` to use the passphrase environment variable name only.
  4. Configure an offsite destination (S3-compatible bucket, rsync target, or cold storage).
  5. Document the destination location and access controls in `.security/SECRETS_INVENTORY.md`.
- verification method: run `./scripts/backup-postgres.sh`, decrypt the resulting `.gpg` file with the passphrase, and restore it to a scratch database

`ACTION REQUIRED FROM OWNER`
- item needed: approve any public release of AI-generated or operator-facing content from `ai_private` or `my_private`
- reason: moving content from private to public buckets changes exposure and may affect privacy or trust
- where it is needed: content approval workflow, public portal publish flow, help center
- exact steps:
  1. Review the content for secrets, PII, and brand alignment.
  2. Move or copy the approved content into the `my_public` or `ai_public` storage path.
  3. Record the approval in an audit log.
  4. Republish the static site or help center.
- verification method: the published content is reachable at the public URL and does not contain secrets when scanned with `truffleHog3`

## Verification

```bash
# Confirm account scoping in the database
docker compose -f compose.prod.yml exec postgres psql -U postgres -d abc_io -c "
SELECT c.id, c.account_id, COUNT(m.id) AS message_count
FROM conversations c
LEFT JOIN direct_messages m ON m.conversation_id = c.id
GROUP BY c.id, c.account_id
ORDER BY c.id
LIMIT 10;
"

# Confirm no secrets in public paths
truffleHog3 --json services/public-portal/src/public --no-history

# Verify backup encryption
ls -lt /backups/postgres/*.sql.gpg | head -n 1
gpg --list-packets /backups/postgres/latest.sql.gpg | grep -i cipher
```

Expected: account-scoped queries return no cross-account rows, public paths contain no secrets, and backups are encrypted.
